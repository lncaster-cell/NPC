# Ambient Life — глубокий технический аудит (Stage I.1)

## 0) Область и ограничения аудита

- Проверка выполнена по текущему коду `scripts/ambient_life` и актуальной документации в `docs/`.
- Скрипты из `third party` **не анализировались** (по вашему требованию).
- Фокус: архитектурная состоятельность, соответствие возможностям NWN2-движка/NWScript, сравнение с распространёнными подходами (FSM/BT/GOAP/utility), риски и практические рекомендации.

---

## 1) Executive summary

Система в текущем виде технически грамотна и хорошо «подогнана» под ограничения NWN2:

1. **Правильная модель исполнения для NWScript**: event-driven + area loop вместо per-NPC heartbeat/polling.
2. **Контроль нагрузки**: tier-модель `FREEZE/WARM/HOT`, токенизированный тик, bounded-очереди действий.
3. **Чёткая декомпозиция подсистем**: route/transition/sleep/reaction разделены и собираются через core dispatcher.
4. **Сильные инварианты безопасности runtime**: проверки валидности объектов/area/tier присутствуют практически во всех entry-path.

Основные системные риски не в «ядре», а в эксплуатационном контуре:

- молчаливое переполнение реестра при `AL_MAX_NPCS`;
- высокая чувствительность к качеству toolset-контента (locals/tag/step);
- отсутствие встроенной телеметрии, из-за чего деградация заметна поздно;
- реактивная ветка I.1 пока локальная и без социально-игровой эскалации (это осознанно, но ограничивает поведение).

## 2) Что в архитектуре сделано сильно

### 2.1 Event-driven runtime, совместимый с движком

Решение отказаться от heartbeat на каждом NPC и свести симуляцию к area-driven loop — это самый правильный выбор для NWScript-окружения с ограниченным скриптовым бюджетом. Тик централизован в области, а события доставляются только зарегистрированным NPC через `OnUserDefined`.

Практический эффект:
- лучше масштабируется по числу NPC;
- проще контролировать частоту и «радиус» вычислений;
- меньше риск неуправляемых циклов.

### 2.2 LOD-политика и bounded-исполнение

Tier-переходы `FREEZE/WARM/HOT`, linked-area warm retention и тик по токену уменьшают лишние вычисления в неактивных зонах. В `HOT`-режиме слот меняется дискретно (`hour/4`) и вызывает только нужный dispatch.

Bounded route progression (через `AL_EVENT_ROUTE_REPEAT`) ограничивает длительность одной рутины, что особенно важно для NWN2 action queue.

### 2.3 Плотный registry и O(1)-удаление

Плотный массив `al_npc_0..N` + swap-remove при удалении/компактации — хорошее инженерное решение под отсутствие удобных динамических структур в NWScript.

Это предсказуемо по памяти и времени и не требует дорогих глобальных сканов.

### 2.4 Корректное разделение ответственности

Система разбита на независимые подсистемы:
- route cache и routine runtime;
- transition runtime;
- sleep runtime;
- blocked/disturbed reactions.

При сбоях в одной ветке используются fallback/reset/resync, что снижает шанс «мертвых» NPC состояний.

## 3) Проверка на техническую согласованность с NWN2/NWScript

### 3.1 Что согласовано хорошо

- Широко применяются guard-проверки: валидность объекта, тип, `!GetIsPC`, соответствие area/tier.
- Используется `DelayCommand` с токеном: старые тики отбрасываются после смены tier/token.
- Действия в рутинах «короткие» и завершаются signal-событием для следующего шага.
- Спец-подсистемы (transition/sleep) не смешаны с обычной активностью — это упрощает дебаг и восстановление.

### 3.2 Точки, где есть технический долг

1. **Нет обязательной диагностической поверхности**:
   - отказ регистрации при переполнении реестра происходит без сигнала;
   - fallback-и (например, route -> idle) не оставляют стандартного следа уровня area/NPC.

2. **Ограниченная observability состояния**:
   - много runtime locals, но нет единого «health snapshot» по area;
   - нет скрипта быстрой валидации контента (на уровне locals/route coverage).

3. **Контент-зависимые сбои маскируются как допустимые fallback-и**:
   - формально безопасно, но эксплуатационно сложно поймать первопричину.

## 4) Сверка с популярными архитектурными подходами

## 4.1 FSM (Finite State Machine)

**Сходство**: текущая система уже содержит FSM-черты (route active, step idx, transition/sleep phase).  
**Отличие**: state хранится преимущественно в locals и продвигается eventами, а не единой централизованной таблицей переходов.

Оценка:
- Для NWScript это нормальный практический компромисс.
- Полный «чистый FSM-движок» дал бы более формальную верификацию, но усложнил бы авторинг и поддержку.

## 4.2 Behavior Tree (BT)

BT популярны для AAA-поведения, но для NWN2 обычно тяжелее в сопровождении:
- выше стоимость интерпретации дерева;
- сложнее обеспечить bounded-выполнение без доп. рантайма;
- toolset-контенту труднее поддерживать BT-структуры.

Текущая event+route модель лучше соответствует целям «стабильность/дешевизна/предсказуемость».

## 4.3 GOAP / Utility AI

GOAP/utility дают гибкое целеполагание, но требуют:
- системы мира/фактов;
- оценки стоимости/полезности;
- планировщика и обширной телеметрии.

Для текущей задачи ambient life это избыточно. Локальные реактивные слои (I.0/I.1/I.2) поверх route-плана выглядят более прагматично.

**Вывод сравнения**: выбранная архитектура оптимальна для NWN2-сценария с большим числом NPC и ограниченным runtime-бюджетом. Улучшать стоит не «парадигму», а эксплуатационные инструменты: диагностику, валидацию контента, контроль деградаций.

## 5) Детальная ревизия принятых решений

### 5.1 Event bus

Событийный диапазон компактен и семантически чист:
- `3100..3105` — смена слота;
- `3106` — resync;
- `3107` — bounded repeat/advance;
- `3108` — blocked resume.

Решение удачное: события простые, минимально связаны, легко трассируются.

### 5.2 Area lifecycle и тики

Сильные стороны:
- token-gating защищает от устаревших тиков;
- tiers не дают симуляции «утекать» в пустых областях;
- linked areas подогреваются ограниченно.

Риск:
- сложнее reasoning при большом количестве linked-area графов (возможны трудные для чтения каскады WARM).

### 5.3 Registry

Сильные стороны:
- компактность и удаление без дыр;
- регулярная компактация защищает от мусора ссылок.

Риск:
- hard cap 100 без метрик/алертов = «тихая» функциональная деградация.

### 5.4 Route cache и routine

Сильные стороны:
- сортировка шагов, проверка на дубликаты и непрерывность `al_step`;
- привязка к текущей area NPC;
- fallback в idle при невалидном маршруте.

Риск:
- без диагностик контент-команда не видит, почему NPC «ушёл в idle».

### 5.5 Transition subsystem

Сильные стороны:
- явные типы переходов;
- строгая проверка endpoint-ов и area-валидности;
- отделённый runtime state.

Риск:
- reliance на корректность tag-ов источника/назначения; ошибки конфигурации часты в production-контенте.

### 5.6 Sleep subsystem

Сильные стороны:
- fallback sleep-on-place при неполных bed waypoint;
- фазовый runtime (`APPROACH/POSE/PLACE`) хорошо читается.

Риск:
- отсутствие автоматической проверки пар `*_approach/*_pose` перед запуском контента.

### 5.7 Blocked / Disturbed reactions

Сильные стороны:
- bounded-логика, без бесконечных ретраев;
- явный return to routine/resync fallback.

Ограничение:
- I.1 сознательно локальный: нет социального распространения тревоги/ролей.

## 6) Рекомендации (приоритетно)

## P0 — обязательно

1. **Ввести минимальную runtime-диагностику (единый toggle)**
   - логировать: отказ `AL_RegisterNPC` (cap), route-cache fail причины, transition/sleep endpoint fail;
   - включение через area/module local, чтобы не шуметь в релизе.

2. **Сделать content preflight validator**
   - проверка `alwp0..5`, непрерывности `al_step`, уникальности шагов, area-consistency, валидности `al_trans_*` и `al_bed_id`-пар.

3. **Сформировать smoke-профиль “dense area”**
   - сценарии: 80/100/120 NPC; частые входы/выходы; blocked/disturbed storm;
   - метрики: доля NPC в idle fallback, resync frequency, dropped registrations.

## P1 — важно

4. **Добавить “health snapshot” по area**
   - `n_registered`, `n_active_route`, `n_fallback_idle`, `n_recent_resync`, `tier`, `slot`.

5. **Стабилизировать контракт linked areas**
   - документировать ограничения на граф линков (например, без «чрезмерной полной связности» для крупных модулей).

6. **Stage I.2 провести как policy-layer поверх I.1**
   - не менять низкоуровневые route/runtime инварианты;
   - добавить только роль/эскалацию (guard/civilian/alarm radius), сохранив bounded scope.

## P2 — улучшения

7. **Операторский runbook деградаций**
   - «NPC не двигается», «массовый idle», «спорадический blocked loop», «disturbed без реакции».

8. **Нормировать naming/namespace для runtime locals**
   - префиксные группы уже хорошие, но полезно добавить таблицу “owner subsystem → ключи → lifecycle”.

## 7) Итоговая оценка

**Общая техническая оценка: высокая (8/10).**

Почему не 10/10:
- эксплуатационная диагностика пока слабее, чем сама архитектура;
- сильная зависимость от дисциплины контент-авторинга без автоматической валидации;
- реактивный слой I.1 ещё не закрывает социальную логику (ожидается I.2).

Если закрыть P0-пакет (telemetry + preflight validator + dense-area smoke), система станет не только архитектурно сильной, но и операционно надёжной для долгой поддержки.
