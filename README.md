# NPC Ambient Life v2

Краткая, читаемая выжимка мастер-плана из `docs/12_MASTER_PLAN.md` с сохранением всех ключевых пунктов: канон архитектуры, границы ответственности, инварианты, runtime-карта, текущий статус и план внедрения.

## Что это за проект

`ambient_life` — модульная система поведения NPC для NWN2: lifecycle, маршруты, переходы между area, суточные активности, сон, реактивные события, crime/alarm и population respawn. Архитектура разделена на слой движка, слой постоянного мира (SQLite/NWNX4) и слой игровой логики, чтобы право/владение/документы не смешивались с мгновенной боевой реакцией ИИ.

Текущий baseline: реализованы стадии A–I.2. Следующий этап — I.3 (legal/reinforcement): усиление guard policy, arrest/trial pipeline, расширенные последствия инцидентов, smoke/QA по новой цепочке.

## Краткий мастер-план по системам

| Система | Что уже зафиксировано (кратко) | Что важно держать в голове (кратко) |
|---|---|---|
| Канон мировой модели (2026-03-14) | Принят обязательный канон: государства, поселения, владельцы, законы, гражданство, титулы, документы, преступления, свидетели и тревога как единая модель мира. | Этот канон имеет приоритет над более ранними решениями по той же области; устаревшие подходы не развиваем. |
| Слои архитектуры | 1) NWN2 engine layer: perception/disturbed/listen/faction reactions. 2) Persistent world layer: SQLite (`xp_sqlite`) как источник истины. 3) Project logic layer: legal/crime/alarm/rights orchestration. | Нельзя подменять персистентные правовые сущности только фракциями движка; каждый слой решает свой класс задач. |
| NWN2 фракции и восприятие | Фракции используются как локальные роли поведения (hostile/neutral/friendly), perception и disturbed — как штатные входы реакции NPC. | Фракции не являются моделью государства/права; `ChangeFaction` не решает юридическую модель для PCs. |
| Персистентный мир (SQLite) | Храним realm/settlement/owner/law/citizenship/titles/rights/documents/diplomacy/criminal flags и связанные профили. | В SQLite хранится «долгая правда мира»; в locals/runtime — только оперативное состояние исполнения. |
| Государства, поселения, владельцы | Поселение — контейнер норм: чья земля, какая стража, какие нормы и документы действуют. Владелец абстрагирован (`REALM`, `PLAYER_ORGANIZATION`, и др.). | Смена владельца/режима — штатный сценарий, а не исключение; модель должна поддерживать политические переходы. |
| Закон и правоприменение | Закон задаётся профилями и режимами (не только bool): убийство/кража/арест и др. с контекстом (власть, дуэль, самооборона, outlaw и т.п.). | Решение «преступление/не преступление» принимается через профиль права поселения и контекст события, не «на глаз». |
| Гражданство, титулы, полномочия | Отдельные сущности для статуса гражданина, титулов, делегируемых прав и jurisdiction. | Права действуют в конкретной юрисдикции и должны проверяться системно перед правовыми действиями. |
| Документы | Документы имеют игровой смысл (мандаты, ордера, пропуска, подтверждения прав) и связаны с проверяемыми правами. | Документ не декоративный: он должен верифицироваться и влиять на legal flow (доступ, арест, санкции и т.д.). |
| Репутация, преступления, свидетели, тревога | Crime pipeline: классификация инцидента, свидетели, распространение сигнала, уровень тревоги области, реакция стражи. | Система событий должна быть ограниченной по нагрузке (без unbounded world-wide scan) и предсказуемой в QA. |
| Runtime-карта `ambient_life` | Покрыты core/area lifecycle, registry/dispatch/cache, route/transition/sleep/activity/schedule, reactive/city, wrappers/hooks, debug/support. | Изменения вносятся по контрактам и инвариантам; не ломаем разделение обязанностей контента и runtime. |
| Контракты данных и событий | Канонизированы locals NPC/Area, sleep/route/population ключи, и внутренний event bus. | Контент обязан поставлять валидные ключи; runtime обязан безопасно обрабатывать fallback без хаотичных hotfix. |
| Population Respawn | Зафиксированы цели, термины, hooks, алгоритмы `OnSpawn`/`OnDeath`, pre-checks RespawnTick, create-path и запреты. | Респаун должен быть контролируемым, без бесконечных циклов и без разрушения perf-гейтов. |
| Operations / Perf / QA | Определены perf-gate, обязательные метрики, операторский checklist, preflight контента, smoke-сценарии. | Любое изменение оценивается через метрики и smoke; «работает локально» без операционных проверок недостаточно. |
| Статус и roadmap | Baseline подтверждён (A–I.2), I.3 запланирован, DoD для I.3 формализован. | Развитие идёт этапами с критериями приёмки; не смешиваем в одном шаге весь «большой симулятор». |
| План восстановления с нуля | Есть практический порядок восстановления подсистем и критерии приёмки каждого шага. | Документ пригоден для онбординга/рекавери: можно последовательно восстановить систему без догадок. |
| Диагностика, FAQ, глоссарий, анти-паттерны | Зафиксированы точки первичной отладки, типовые ответы, термины, запреты на вредные подходы. | Это рабочие guardrails: сначала проверяем канонические точки, не внедряем shortcut-решения в обход архитектуры. |
| Полнота и поддержание документации | Выполнена формальная проверка покрытия: документация, runtime-файлы, сущности, сценарии, planned-часть и границы области. | Master-документ поддерживается как единая точка входа; README — краткий навигационный слой к нему. |

## Границы ответственности

**Контент отвечает за:** корректные локалы, точки маршрутов/сна/активностей, конфиги area/population, и соблюдение контрактов.

**Runtime отвечает за:** lifecycle, реактивные переходы, устойчивость к сбоям, ограничение нагрузки, правовой и событийный пайплайн по утверждённым инвариантам.

## Что запрещено архитектурно

- Подменять правовую/политическую модель только engine-фракциями.
- Делать unbounded-сканирования мира для legal/reinforcement/population задач.
- Ломать контракты locals/events и «чинить» это ad-hoc правками без системного решения.
