# AI research digest: ограничения движка и практические рекомендации

Этот файл фиксирует полезные выводы из исследовательского черновика `вф` и оставляет только то, что имеет практическую ценность для текущего reset/rebuild цикла.

## 1) Архитектурный вектор
- Базовая модель исполнения: **area-centric + event-driven orchestration** вместо частых per-NPC полных проходов.
- Любая подсистема должна работать в режиме **bounded runtime** (бюджет тиков/очередей/операций на цикл).
- При перегрузке: **degrade gracefully** (defer, drop low-priority, fallback), а не пытаться выполнить всё в одном цикле.

## 2) Ограничения NWN2/NWScript, которые учитывать по умолчанию
- Тяжёлые циклы и массовые сканы объектов в heartbeat-путях недопустимы для масштабной симуляции.
- Состояние долгой жизни и агрегаты мира лучше хранить вне «летучей» логики скриптов — в персистентном слое.
- Интеграция должна опираться на событие-ориентированные точки входа и явные контракты данных между доменами.

## 3) Рекомендованная форма библиотеки
- Слоистая платформа:
  1. **Core runtime**: dispatch queue, budgets, priorities, backpressure.
  2. **Domain modules**: daily life, city response, legal, property, travel, trade/city state, clan, aging.
  3. **Persistence adapters**: SQLite/NWNX4 как SoT для долгоживущих данных.
- Доменные модули должны быть связаны через **явные события и схемы данных**, а не через скрытые зависимости.

## 4) Контент и синхронизация
- Поведение NPC (route/routine/transition) по возможности хранить как **данные**, а не как монолитный код.
- Добавить обязательные поля синхронизации контента:
  - `content_version`
  - `fingerprint/hash`
  - `migration_required` (или эквивалент)
- Любой кэш маршрутов/рутин должен иметь правила инвалидирования по версии контента.

## 5) Производительность и эксплуатация
- Ввести минимальный набор эксплуатационных метрик:
  - queue depth
  - average/peak dispatch time
  - dropped/deferred events
  - cache hit ratio
- Для каждого домена определить SLO на цикл и деградационные режимы.

## 6) Практический baseline для реализации
- Начинать с **модульной event-driven базы** (наиболее совместимо с текущим docs-first каноном проекта).
- ECS-подобные приёмы использовать точечно там, где есть выгода от батч-агрегации.
- Мини-VM/DSL для поведения применять только в зонах с высокой контентной вариативностью и стабильным tooling.

## 7) Что не переносить из черновика как «жёсткий факт»
- Внешние утверждения без верификации в коде/доках проекта.
- Конкретные численные лимиты, не подтверждённые профилированием на целевом сервере.
