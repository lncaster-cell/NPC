# 45 — Daily Life v2 Log Removability Rule (RU)

> Дата: 2026-04-08  
> Статус: approved working rule for implementation

## 1) Цель

Зафиксировать, что smoke/debug-логирование в Daily Life v2 является вспомогательным диагностическим слоем,
а не частью обязательной бизнес-логики runtime.

## 2) Базовое правило

- Логи должны быть устроены так, чтобы их можно было:
  - отключить флагом,
  - локально удалить,
  - централизованно вырезать helper-слой,
  без изменения поведенческого контракта рабочей функции.

## 3) Обязательное требование

Удаление логов после стабилизации функции не должно:
- менять результат работы функции;
- менять порядок её бизнес-ветвления;
- менять состояние runtime-данных;
- ломать smoke/runtime-контур, кроме исчезновения самой диагностики.

## 4) Архитектурное следствие

Поэтому логовый слой должен быть:
- отделён от core-логики;
- подключён через helper/adapter;
- без критической зависимости gameplay-кода от `SendMessageToPC(...)`;
- без хранения обязательного runtime-state только внутри logging-вызовов.

## 5) Что запрещено

Недопустимо:
- завязывать полезную логику функции на факт отправки сообщения в чат;
- рассчитывать на logging-вызов как на часть control-flow;
- смешивать изменение state и отладочный вывод в одном неразделимом блоке;
- писать код так, чтобы удаление логов меняло семантику исполнения.

## 6) Практическое правило для разработки

Любой временный smoke/debug-log должен добавляться так,
чтобы в будущем его можно было безопасно вырезать отдельным микро-шагом:
- либо удалением вызова helper;
- либо удалением/отключением helper-реализации;
- либо полным удалением временного smoke-скрипта.

## 7) Definition of Done для логов

Логовый слой считается реализованным корректно, только если одновременно выполняется:
1. он пишет в чат только как диагностический слой;
2. он выключен по умолчанию;
3. он не создаёт заметной нагрузки при выключении;
4. его можно безопасно удалить без изменения runtime-поведения.
