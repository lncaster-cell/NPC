# 44 — Daily Life v2 Chat Logging Policy (RU)

> Дата: 2026-04-12  
> Статус: synced to current implementation (temporary debug-phase policy)

## 1) Назначение

Зафиксировать единое правило логирования для Daily Life v2 на текущем debug/owner-run этапе и убрать недопонимание между policy-текстом и фактическим кодом.

## 2) Базовое правило

- Smoke/debug-логи Daily Life v2 пишутся в игровой чат.
- Runtime-логирование централизовано через helper `DL_LogRuntime`.
- Chat-лог используется как быстрый канал наблюдения за результатом проверки функции.
- Формат сообщений должен оставаться коротким и машинно читаемым.

## 3) Текущее фактическое поведение в `main`

Сейчас в коде:
- chat-output контролируется модульным флагом `dl_chat_log`;
- вызовы `SendMessageToPC(...)` выведены в централизованный helper `DL_LogRuntime`;
- `PrintString(...)` остаётся частью текущего debug-oriented runtime path;
- на текущем debug-этапе `DL_InitModuleContract()` инициализирует `dl_chat_log` во включённое состояние при первом запуске модуля.

Это сделано как временный режим owner/toolset observability, а не как финальная performance-hardening политика.

## 4) Целевая hardening-позиция (не финализирована в коде)

Целевое направление после завершения текущей debug-итерации:
- расширенные smoke/debug chat-логи должны быть выключены по умолчанию;
- логи должны включаться только явным флагом диагностики;
- при выключенном chat-debug нежелательно держать постоянный chat-spam в noisy hooks.

Пока эта целевая позиция ещё не доведена до полного code parity и не должна трактоваться как уже реализованный факт.

## 5) Требование по производительности

Текущее правило:
- не плодить разрозненные прямые `SendMessageToPC(...)` по коду;
- держать логирование централизованным;
- не расширять debug-output без owner/debug необходимости.

Целевой hardening после текущей итерации:
- уменьшить overhead при выключенном chat-debug;
- не держать постоянный debug-chat включённым в штатном runtime.

## 6) Управление

Текущая модель управления:
- модульный флаг `dl_chat_log` — для игрового chat-output;
- централизованный helper `DL_LogRuntime` вместо разрозненных прямых вызовов.

## 7) Практическое следствие для разработки

Новая функция может временно получать chat-log для отладки и smoke-проверки,
но текущий статус нужно понимать честно:
- централизованный helper уже внедрён;
- chat-output сейчас временно живёт в debug-oriented режиме;
- после стабилизации этот режим может быть ужесточён до disabled-by-default без изменения архитектурной роли helper-а.

## 8) Решение для следующих шагов

При дальнейшей реализации Daily Life v2:
1. не возвращаться к разрозненным прямым вызовам chat-log по коду;
2. сохранять централизованный helper как единую точку runtime-логирования;
3. трактовать текущий default-on chat-debug как временный debug-режим, а не как окончательную политику;
4. после завершения текущего owner/debug цикла отдельно решить, когда переводить chat-output в strict disabled-by-default режим.
