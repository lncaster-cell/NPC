# 44 — Daily Life v2 Chat Logging Policy (RU)

> Дата: 2026-04-08  
> Статус: approved working rule for implementation

## 1) Назначение

Зафиксировать единое правило логирования для Daily Life v2 на этапе bootstrap / smoke / debug.

## 2) Базовое правило

- Smoke/debug-логи Daily Life v2 пишутся в игровой чат.
- Chat-лог используется как основной быстрый канал наблюдения за результатом проверки функции.
- Формат сообщений должен быть коротким и машинно читаемым, например:
  - `[DL2][SMOKE][STEP01][PASS] ...`
  - `[DL2][DEBUG][SCOPE] ...`
  - `[DL2][ERROR][SCOPE] ...`

## 3) Правило по умолчанию

- Любые расширенные smoke/debug chat-логи должны быть **выключены по умолчанию**.
- Логи включаются только явным флагом диагностики.
- Отсутствие включённого флага должно означать отсутствие chat-spam и отсутствие заметной runtime-нагрузки.

## 4) Требование по производительности

При выключенных логах допустим только минимальный overhead:
- чтение флага;
- одна ранняя проверка условия;
- немедленный `return` без сборки длинных строк и без отправки сообщений в чат.

Недопустимо:
- собирать длинное сообщение до проверки флага;
- спамить чат в noisy/high-frequency hooks;
- держать постоянный debug-chat включённым в штатном runtime.

## 5) Разделение типов логов

### 5.1 Smoke log
Используется для точечной проверки конкретного микро-шага или функции.

Свойства:
- может быть шумным;
- живёт только на время отладки/проверки;
- должен отключаться централизованно.

### 5.2 Runtime log
Используется только для редких и важных диагностических событий.

Свойства:
- должен быть существенно реже smoke-log;
- не должен превращаться в постоянный flood;
- не должен нарушать budget runtime-контура.

## 6) Управление

Рекомендуемая модель управления:
- модульный флаг вида `dl2_smoke_trace` — для smoke-log;
- отдельный флаг/уровень логирования для runtime-debug при необходимости;
- централизованный helper логирования вместо прямых `SendMessageToPC(...)` по всему коду.

## 7) Практическое следствие для разработки

Новая функция может временно получать chat-log для отладки и smoke-проверки,
но после стабилизации:
- шумный smoke-log должен отключаться флагом,
- или удаляться как временный диагностический слой,
- при этом базовый runtime не должен зависеть от постоянного chat-log.

## 8) Решение для следующих шагов

При дальнейшей реализации Daily Life v2:
1. не использовать разрозненные прямые вызовы chat-log по коду как постоянный паттерн;
2. вынести логирование в единый helper;
3. держать расширенную диагностику disabled-by-default;
4. использовать chat-log прежде всего как инструмент smoke/debug-валидации.
