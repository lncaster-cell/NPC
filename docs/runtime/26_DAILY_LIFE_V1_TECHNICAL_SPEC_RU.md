# Ambient Life v2 — Техническое задание (ТЗ) на дальнейшую разработку Daily Life v1

Дата: 2026-04-02  
Статус: draft for execution  
Назначение: формализовать работы следующего этапа разработки **на основе действующей проектной документации и текущего состояния кода**.

---

## 1. Основание для ТЗ

ТЗ сформировано по следующим документам проекта:
- `docs/canon/17_UNIFIED_GAME_DESIGN_BRIEF_RU.md` — единый канон проекта;
- `docs/canon/12B_DAILY_LIFE_VNEXT_CANON.md` — канон Daily Life vNext;
- `docs/runtime/12B_RUNTIME_MASTER_PLAN.md` — runtime-инварианты и roadmap;

- `docs/runtime/12B_DAILY_LIFE_V1_IMPLEMENTATION_STATE.md` — зафиксированное текущее состояние реализации.

---

## 2. Контекст и текущая стадия

### 2.1 Что уже реализовано (кодовый baseline)
Реализован каркас Milestone A (Steps A–E) в `scripts/daily_life/`:
- контракты и utility/logging;
- deterministic resolver с override (`FIRE`, `QUARANTINE`);
- materialization + activity/interaction;
- area lifecycle (`HOT/WARM/FROZEN`) и bounded worker;
- slot-handoff stub для внешнего слоя population.

### 2.2 Что ещё не закрыто
Milestone A не может считаться завершённым до фактического закрытия acceptance по сценариям A–G на owner/toolset-среде с фиксацией verdict в журнале приёмки.

---

## 3. Цель этапа по этому ТЗ

### 3.1 Главная цель
Довести Daily Life v1 до **операционно подтверждённого** состояния Milestone A и подготовить безопасный переход к следующему этапу интеграций без расширения scope сверх канона.

### 3.2 Измеримый результат
Результатом этапа считается:
1. подтверждённая работа runtime-контура в HOT/WARM/FROZEN по сценариям приёмки;
2. заполненный acceptance journal с финальными статусами (`PASS/PARTIAL/FAIL`) и комментариями по расхождениям;
3. закрытый список критических расхождений Milestone A в коде/контрактах;
4. зафиксированный readiness-report для перехода к следующему milestone.

---

## 4. Предмет работ (In Scope)

### 4.1 Приёмка Milestone A (обязательно)
1. Выполнить smoke/acceptance прогоны A–G по runbook.
2. Зафиксировать результаты в журнале приёмки.
3. Для каждого `PARTIAL/FAIL` создать карточку расхождения:
   - условие воспроизведения;
   - ожидаемое поведение;
   - фактическое поведение;
   - причина (код / контент / данные area / контракт);
   - решение и срок.

### 4.2 Точечные доработки runtime в рамках A
Разрешены только изменения, нужные для устранения расхождений Milestone A:
- корректность resolver при временных окнах и override;
- устойчивость materialization fallback-цепочек;
- стабильность bounded-worker budget и resync-причин;
- консистентность interaction-state (`dialogue_mode`, `service_mode`).

### 4.3 Синхронизация документации с кодом
Обязательно обновить runtime-truth заметки там, где термины/enum-ы расходятся между spec и Milestone A-реализацией.

---

## 5. Что НЕ входит в этап (Out of Scope)

Запрещено включать в текущий этап:
- полноценную population/respawn систему beyond stub-handoff;
- legal/trial/court pipeline как новый контур;
- macro trade/city-state экономику;
- travel/clan/aging интеграции;
- расширение набора директив/окон сверх необходимого для закрытия A;
- любые world-wide тяжёлые сканы и небюджетные loop-решения.

---

## 6. Нормативные требования к реализации

### 6.1 Архитектурные требования
1. Сохранить area-centric + event-driven модель.
2. Соблюдать bounded processing (budget, caps, predictable dispatch).
3. Heartbeat использовать как gate, не как главный per-NPC loop.
4. Не смешивать Daily Life runtime с legal truth и другими доменами.

### 6.2 Требования к area-tier
- `HOT`: полный локальный runtime Daily Life и materialization;
- `WARM`: только ограниченный подготовительный контур;
- `FROZEN`: отсутствие живого runtime-исполнения.

### 6.3 Требования к качеству контрактов
- кодовые константы Milestone A считаются source-of-truth для runtime;
- документация должна явно помечать несовпадения с vNext-терминологией;
- доступ к NPC/area-полям идёт через helper-слой, а не произвольным чтением locals в разных модулях.

---

## 7. Требования к тестированию и приёмке

### 7.1 Обязательные группы проверок
1. **Lifecycle-проверки:** переходы HOT↔WARM↔FROZEN, отсутствие лишней активности в FROZEN.
2. **Resolver-проверки:** корректная директива по времени/профилю/override.
3. **Materialization-проверки:** корректный anchor выбор, fallback без зависаний.
4. **Worker-проверки:** budget-bound обработка без перегрузки и бесконечных resync.
5. **Interaction-проверки:** корректность `dialogue_mode` / `service_mode` после пересчёта.

### 7.2 Критерии готовности этапа (Definition of Done)
Этап считается завершённым только если:
- все обязательные acceptance-сценарии пройдены либо имеют согласованные non-blocking исключения;
- нет открытых блокирующих дефектов уровня Critical/High в рамках Milestone A;
- зафиксирован финальный owner verdict по готовности.

---

## 8. Артефакты этапа

По завершении должны быть подготовлены:
1. Обновлённый код `scripts/daily_life/*` (только в границах A).
2. Обновлённый acceptance journal с фактическими результатами.
3. Краткий inspection report по исправленным расхождениям.
4. Readiness note: «Milestone A closed / not closed», с причинами.

---

## 9. Риски и ограничения

1. **Риск контент-данных area:** ошибки разметки anchors/base могут маскироваться под кодовые дефекты.
2. **Риск терминологического дрейфа:** vNext-термины и runtime-имена Milestone A могут расходиться.
3. **Риск scope creep:** попытка включить legal/trade/population расширения сорвёт сроки закрытия A.

Митигирующие меры:
- фиксировать причину дефекта (код/контент/контракт) отдельно;
- вести changelog соответствий «doc-name ↔ runtime-name»;
- отклонять любые задачи вне In Scope.

---

## 10. План выполнения (порядок работ)

1. **Подготовка прогона:** актуализировать runbook, тестовые area, флаги трассировки.
2. **Прогон A–G:** выполнить полную серию acceptance-сценариев.
3. **Анализ:** классифицировать найденные расхождения и приоритизировать.
4. **Исправления:** внести минимально необходимые правки в runtime.
5. **Репрогон:** подтвердить устранение расхождений.
6. **Фиксация:** обновить журналы/отчёты и вынести финальный verdict.

---

## 11. Правило управления изменениями

Любое новое требование, не нужное для закрытия Milestone A, оформляется как отдельная задача следующего этапа и не включается в это ТЗ без явного решения владельца проекта.
