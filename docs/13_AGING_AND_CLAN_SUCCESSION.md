# Система старения и смерти персонажа через клан

Дата: 2026-03-14  
Статус: design-блок для реализации v1

---

## 1) Цель

Система должна:

- делать жизнь персонажа конечной;
- сохранять долгосрочный прогресс игрока через клан, а не через «бессмертного» героя;
- работать с минимальной серверной нагрузкой;
- использовать встроенный возраст NWN2 только как стартовую величину.

Канон: **персонаж смертен, клан постоянен**.

---

## 2) Правила

### 2.1. Источники возраста

Используются два возраста:

1. **Встроенный возраст NWN2 (`GetAge`)** — читается один раз при первом входе и записывается как `base_age`.
2. **Игровой возраст системы** — далее считается серверной логикой по календарю мира.

Важное ограничение движка: есть `GetAge()`, но штатного `SetAge()` нет.

### 2.2. Календарь мира

Возраст считается от календаря модуля:

- `GetCalendarDay`
- `GetCalendarMonth`
- `GetCalendarYear`
- `SetCalendar`

Для NWN2 фиксируется календарный формат:

- месяц: `1..12`
- день: `1..28`
- `SetCalendar` двигает время только вперёд.

### 2.3. Производительная формула

Без heartbeat и фоновых таймеров.

Абсолютный день мира:

`abs_day = year * 336 + (month - 1) * 28 + (day - 1)`

Возраст персонажа:

`current_age = base_age + ((current_abs_day - birth_abs_day) / 336)`

---

## 3) Поток (v1)

### 3.1. На первом входе персонажа

1. Считать `base_age = GetAge(oPC)`.
2. Зафиксировать `birth_abs_day` из текущего календаря.
3. Определить расу персонажа.
4. Один раз сгенерировать `death_age` по расовому диапазону.
5. Сохранить в БД.

### 3.2. На каждом входе в модуль

1. Загрузить запись персонажа из БД.
2. Пересчитать `current_age` по формуле.
3. Проверить предупреждения о старости.
4. Если `current_age >= death_age`, зафиксировать смерть от старости.

Проверка старения в v1 выполняется **только при входе в модуль**.

---

## 4) Данные

Обязательные поля персонажа в БД:

- `character_id`
- `clan_id`
- `base_age`
- `birth_abs_day`
- `death_age`
- `old_age_warn_stage`
- `is_dead`
- `death_reason`

Семантика ключевых полей:

- `base_age` — стартовый возраст, выбранный при создании;
- `birth_abs_day` — абсолютный день, в который зафиксирован старт;
- `death_age` — персональный порог смерти;
- `old_age_warn_stage` — стадия предупреждений;
- `death_reason` — причина смерти (`OLD_AGE`).

---

## 5) Расовые диапазоны смерти

Игровая серверная таблица (кастомная, более суровая):

- Человек: `50–85`
- Дварф: `80–120`
- Эльф: `200–240`
- Полуорк: `40–60`

`death_age` генерируется один раз на персонажа и больше не пересчитывается.

---

## 6) Предупреждения и смерть

### 6.1. Стадии предупреждений

- осталось 5 лет → stage `1`
- осталось 2 года → stage `2`
- остался 1 год → stage `3`

`old_age_warn_stage` предотвращает повторную выдачу одного и того же предупреждения.

### 6.2. Правило смерти

Если при входе:

`current_age >= death_age`

то персонаж помечается умершим:

- `is_dead = 1`
- `death_reason = OLD_AGE`

Для v1 смерть детерминированная (без вероятностного «доживания после порога»).

---

## 7) Клан и наследование

После смерти персонажа:

- клан сохраняется;
- клановые активы сохраняются;
- игра продолжается за наследника клана.

Канон передачи вещей:

- наследование вещей между поколениями допускается **только** через клановое хранилище;
- личное имущество умершего не переносится автоматически, если заранее не переведено в клановую собственность.

Канон прогрессии:

- клан развивает не только текущего персонажа, но и будущих наследников;
- новый персонаж может стартовать с уровнем/опытом, заданным клановой подготовкой.

---

## 8) Ограничения

Система специально построена с учётом ограничений NWN2:

- `GetAge()` есть, `SetAge()` нет;
- календарь мира — отдельная сущность;
- календарь подходит как стабильный источник серверного возраста.

Следствие: старение реализуется проектной логикой поверх календаря, а не надеждой на встроенное автоматическое старение.

---

## 9) Антиабуз

Проектные запреты:

- нет повторного «переброса» `death_age` через релог;
- нет пересчёта `death_age` после первого сохранения;
- нет зависимости возраста от heartbeat/таймеров;
- нет автоматического переноса личного инвентаря умершего в наследство.

Это минимизирует нагрузку, упрощает поддержку и закрывает основные лазейки абуза для v1.
