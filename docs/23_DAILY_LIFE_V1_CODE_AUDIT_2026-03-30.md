## 1) Краткий вердикт

Текущая реализация соответствует модели **rule-driven Milestone A**: resolver/materialize/worker/lifecycle контур присутствует и читается в коде.  
Главный операционный риск: поведение tier lifecycle и фактический запуск worker сейчас расходятся по ожиданиям `WARM`-слоя.

---

## 2) Что подтверждено как сильная сторона

1. **Ясный resolver pipeline**
   - `schedule/day/override -> directive`,
   - `directive -> anchor/dialogue/service`.
   Это соответствует заявленной архитектуре v1.

2. **Явные fallback-ветки для base-lost**
   - recovery через function slot profile,
   - provisional base,
   - затем controlled fallback (`ABSENT/UNASSIGNED`).

3. **Bounded worker и фильтрация NPC**
   - worker budget есть,
   - обрабатываются только Daily Life NPC,
   - приоритет на `named/persistent` или pending-resync.

## 3) Найденные расхождения / риски

### A-01 (высокий): `WARM` budget объявлен, но runtime фактически HOT-only

**Наблюдение:**
- В константах есть бюджеты `HOT/WARM/FROZEN`.
- Но `DL_ShouldRunDailyLife` возвращает `TRUE` только для `HOT`.
- Следовательно, worker прекращает работу в `WARM`, и `DL_BUDGET_WARM` не используется.

**Риск:**
- Документарное ожидание «WARM как подготовительный слой» может восприниматься как активная ограниченная обработка, но текущий runtime этого не делает.

**Рекомендация (без перепридумывания механик):**
- либо синхронизировать документы под HOT-only,
- либо сделать минимальную правку gate-логики для целевого `WARM` поведения (только после owner-решения).

### A-02 (высокий): переход в `WARM` при наличии игроков в зоне на `OnExit`

**Наблюдение:**
- На `OnExit`: если игроки ещё есть в зоне, вызывается `DL_OnAreaBecameWarm`.
- При текущем HOT-only worker gate это останавливает полноценный Daily Life даже когда в зоне всё ещё есть игрок.

**Риск:**
- Непредсказуемая «заморозка» поведения при частичном выходе игроков из мультиплеерной зоны.

**Рекомендация:**
- зафиксировать owner-решение, должен ли remaining-player сценарий держать `HOT`.

### A-03 (средний): возможный дрейф между operational docs и runtime

**Наблюдение:**
- В операционных документах регулярно упоминается различие `HOT/WARM/FROZEN`.
- Без явной оговорки про HOT-only gate это легко трактуется как «WARM тикает ограниченно».

**Рекомендация:**
- держать отдельный Runtime Truth файл и обновлять его при каждой логической правке.

## 6) Продолжение аудита (Phase 2, 2026-03-30)

Ниже — дополнительный проход после первичного отчёта, с фокусом на эксплуатационные edge-cases.

### A-04 (средний): `OnExit` + `DL_HasAnyPlayers` может давать пограничный эффект

**Наблюдение:**
- В обработчике выхода логика tier зависит от `DL_HasAnyPlayers(oArea)`.
- На стороне NWN2 порядок обновления набора объектов в area на событии выхода может давать нюанс: exiting player ещё виден в area в момент проверки.

**Риск:**
- Потенциально нестабильный переход `WARM/FROZEN` на «последнем выходящем» игроке.

**Рекомендация:**
- добавить в smoke/manual сценарий явную проверку «последний игрок покинул area -> ожидаем FROZEN» и зафиксировать фактическое поведение в acceptance.

### A-05 (средний): скрытая цена полного area-scan при worker dispatch

**Наблюдение:**
- Worker ограничивает количество **обрабатываемых** NPC по budget,
  но сам проход делает `GetFirstObjectInArea/GetNextObjectInArea` до исчерпания budget или конца списка.

**Риск:**
- На перегруженных area стоимость сканирования может стать заметной, даже если budget малый.

**Рекомендация:**
- для Milestone A: оставить как есть, но добавить perf-note в operational tracking;
- post-Milestone A: рассмотреть индекс/очередь Daily Life NPC для уменьшения scan-cost.

### A-06 (низкий/операционный): scope ресинка захватывает всех non-PC существ

**Наблюдение:**
- `DL_RequestAreaResync` ставит pending-флаги всем non-PC creatures, не только Daily Life NPC.
- Неформально это безопасно (фильтр позже есть), но создаёт лишний шум состояния.

**Риск:**
- Диагностика становится менее чистой на сценах с большим количеством служебных/временных существ.

**Рекомендация:**
- рассмотреть мягкую фильтрацию уже на этапе `DL_RequestAreaResync` после закрытия Milestone A.

## 9) Минимальные правки после аудита (2026-03-30)

### P-01 (выполнено): синхронизация worker gate с tier-budget контрактом

**Что изменено:**
- `DL_ShouldRunDailyLife` теперь разрешает выполнение worker в `HOT` и `WARM` tier.

**Почему это безопасно:**
- Логика не перепридумана: используется уже существующий бюджетный контракт (`HOT=6`, `WARM=2`, `FROZEN=0`).
- `FROZEN` по-прежнему полностью отключает dispatch.

**Эффект:**
- `DL_BUDGET_WARM` перестал быть «мертвым» параметром;
- сценарий `OnExit` при remaining players больше не переводит area в фактический «runtime stop».
