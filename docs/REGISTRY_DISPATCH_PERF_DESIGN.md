# Registry/Dispatch и bounded-производительность — краткий диздок

## 1) Цель
Стабильно обрабатывать события для большого числа NPC, сохраняя bounded latency и контролируемую нагрузку.

## 2) Канон
- Реестр NPC area-scoped, с ограничением вместимости.
- Массовые area-события идут через queue + batched dispatch.
- Любой overflow/pressure обязан быть диагностируемым через locals/метрики.

## 3) Принципы
- Быстрый доступ через индекс/кэш, fallback только как safety-net.
- Любые операции compact/swap/update обязаны сохранять консистентность reverse-index.
- Drain очереди должен быть прогнозируемым по тикам.

## 4) Ограничения
- Нельзя обходить очередь «прямыми» массовыми вызовами на сотни NPC.
- Нельзя убирать защитные cap/overflow-ограничители.
- Нельзя внедрять новые тяжёлые сканы вместо registry-операций.

## 5) Сценарии
- A: штатная нагрузка S80/S100 -> очередь дренируется в целевых окнах.
- B: стресс S120 -> возможен рост pressure, но без unbounded деградации.
- C: частичные рассинхроны индексов -> fallback + самовосстановление + диагностика.
